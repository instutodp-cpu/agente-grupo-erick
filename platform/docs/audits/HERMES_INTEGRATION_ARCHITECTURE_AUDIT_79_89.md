# Hermes Agent Core — Integration Architecture Audit (PRs #79–#89)

Audit branch: `audit/hermes-agent-core-integration-79-89`. Base: `main` at merge of PR #89 (`5163138`). This is a diagnostic PR only — no production behavior, contract, enum, decision, or fingerprint was changed to produce this report. No Agent Orchestrator was implemented.

> **Update (PR #91):** `CAND-REG-01` (below) has been fixed. See `platform/docs/audits/HERMES_REGISTRY_FINGERPRINT_CONFLICT_FIX.md` for the fix details, files changed, and test coverage. The finding is left below exactly as originally reported, annotated `Status: RESOLVED in PR #91`, so this document remains an accurate historical record of the PR #90 audit.

## 1. Resumo Executivo

PRs #79–#89 (Agent Core, Agent Policy, Agent Session, Agent Memory, Model Provider, Model Selection, the PR85 architecture audit + PR86 fixes, Context Assembly, Tool Contracts, Workflow Contracts) were audited end-to-end for whether their contracts and boundaries can be safely connected by a future Agent Orchestrator: 192 files under `src/core/` (90 in the audited PR #79–89 domains), 68 test files (1303 tests, up from a pre-audit baseline of 1283), 47 fixtures, and 9+ domain docs.

**No P0 finding. No P1 finding.** The core simulation invariant — nothing across all 9 domains can ever claim to have actually executed — was proved by live adversarial probing (not assumed from reading code): every one of the 8 decision/result builders and all 9 audit builders correctly forces every one of ~24 forbidden execution-shaped flags `false`/`true` as appropriate, under a combined override attack, with zero bypasses found; every registry independently re-validates and rejects a hand-crafted malicious record bypassing its own decision builder. This audit also proved a full, exclusively-declarative composition — Agent → Policy → Session → Memory Reference → Model Selection → Context Assembly → Tool Reference → Workflow Reference — using only real, unmodified code across 10 synthetic scenarios (NO_LLM, economical model, declarative workflow, declarative tool, policy blocked, tenant mismatch, organization mismatch, fingerprint mismatch, budget blocked, required approval blocked), all now permanent regression tests. Zero circular dependencies and zero contract-imports-engine/registry violations were found across all 192 files (both the 90 in-scope files and, independently, the full codebase including the 65-file transcription domain, which remains cleanly decoupled from PRs #79–89 in both directions).

That said, the audit found **7 P2 findings** that are real integration-boundary gaps a future Orchestrator must account for — not contract bugs, since every one is a deliberate, correctly-scoped consequence of each domain's "no automatic registry consultation, no implicit cross-domain validation" design:

- The real `ModelSelectionDecision` cannot be mapped onto the `ModelSelectionDecisionReference` PR #87's Context Assembly expects by direct field copy — three field names differ and no single field named `decision_fingerprint` exists on the real decision object at all. An explicit adapter is required and does not exist today (INTEG-01).
- Neither `human_review_required` (Model Selection) nor `approval_required` (Workflow) ever surfaces as a distinct decision `status` — both are boolean metadata on an upstream object the Orchestrator must separately retain and inspect (INTEG-02, INTEG-03).
- Nothing today cross-validates a Workflow step's `required_capabilities` against its referenced Tool's actual `CapabilitySet`, nor a referenced Tool's `side_effect` (e.g. `IRREVERSIBLE_REFERENCE`) against the step's `approval_required`/`compensation_reference` — both are exactly the safety nets the future Orchestrator exists to add (INTEG-04, INTEG-05).
- `agent-registry.js`/`agent-policy-registry.js` (the two oldest, already flagged once in the PR85 audit for a different gap) still have no `FINGERPRINT_CONFLICT`/`expected_fingerprint` support, unlike the other 7 registries (CAND-REG-01).
- Across the 8 decision-bearing domains, the "was this allowed" boolean signal (`allowed_in_simulation`/`eligible_in_simulation`) is present under two different names in 4 of 8 files and entirely absent in the other 4 — an Orchestrator must rely on `status`/`decision` alone, never this field, as the portable cross-domain signal (INTEG-12).

Additionally: **12 P3** findings (mostly naming/shape divergence across domain generations — `effect` vs `decision`, `registry_version`/`logical_sequence` presence patterns, two different "list by organization" calling conventions, `blockers`/`reason_codes`/`logical_sequence` presence asymmetry across the 9 audit modules) and **5 INFO** findings (2 of them positive: `TOOL_CAPABILITIES` is directly shared — not duplicated — between Tool and Workflow with zero drift risk, and `context_assembled`/`content_loaded`/`prompt_generated` are structurally, not just documentedly, impossible to return `true`).

**Recommendation: GO WITH FIXES.** See §19 for the full criteria — the short version is that no P0/P1 exists anywhere in the composed system, and the three items that matter (the INTEG-01 adapter, the two Tool/Workflow cross-checks, the approval/human-review aggregation step) are exactly the Orchestrator's own core responsibilities per the companion readiness document, not fixes to PRs #79–89 themselves.

**Could a single Orchestrator interpret all 8 with one small adapter, or does each need bespoke handling?** Bespoke handling is required for 3 specific reasons, all confirmed above: (1) Agent Policy alone uses `effect` where all 7 others use `decision` — a uniform "what was decided" reader must special-case this one file; (2) the "was this allowed in simulation" boolean exists under 2 different names in 4/8 files and doesn't exist at all in the other 4 — an Orchestrator must rely on `status`/`decision` alone as the portable signal, never on this boolean; (3) fingerprint field names and counts differ per domain by design (richer domains have more granular fingerprints) — an Orchestrator wanting "the one fingerprint for this decision" must know each domain's own field name, there is no portable `decision_fingerprint`-style field across all 8. None of these 3 are contract defects — they reflect real, deliberate architectural evolution across a year-plus of PRs — but they are exactly the kind of "incompatibilidade que pode causar decisão incorreta" the audit spec asks about if an Orchestrator author assumed field-name uniformity without checking.

**`status` vs `decision` shape parity confirmed uniform:** all 8 files show the same two-value discipline on the "outcome" field (an allow-equivalent enum value, or `BLOCKED`/`DENY`), and all 8 correctly force the sentinel/blocked-shape fields (`null` selected-ids, sentinel reference-ids, empty arrays as applicable) whenever `status` isn't the success value — confirmed by this audit's own adversarial-override test plus direct reading of every `build*`'s degrade branch.

## 9. Fingerprints

Confirmed by direct grep: **every one of the 29 files across the 3 newest domains** (`context-assembly-*.js`, `tool-*.js`, `workflow-*.js`) imports `stablePayload`/`cloneFrozen` from `agent-identity-contract.js` — zero local reimplementation of canonicalization anywhere in PRs #87–89. This extends the PR85 audit's PR #79–84 finding cleanly: "fingerprint" remains a canonical, key-sorted JSON serialization (never a cryptographic hash) uniformly across all 9 in-scope domains, confirmed again by this session's own extensive direct use of `stablePayload` while implementing PRs #87–89.

**Array handling:** every array field that participates in a fingerprint across the 3 newest domains is pre-sorted via `uniqueSorted()` or an equivalent ordered-unique-list validator before being fingerprinted (`section_fingerprints`/`source_fingerprints` in `context-assembly-result.js`, `capabilities` in `tool-decision.js`, `step_fingerprints` in `workflow-decision.js`) — consistent with the PR85 audit's finding that array order is preserved (not sorted) by `stableCanonicalize` itself, so every caller needing order-independence must pre-sort before fingerprinting, and all 3 newest domains correctly do.

**Caller-supplied vs. independently-computed fingerprints:** no confusion found — every decision/audit builder in all 9 domains computes its own fingerprint via `stablePayload(actualObject)` at build time; no code path in the 3 newest domains treats an inbound opaque `*_fingerprint` string as something to be re-derived or verified against a recomputation. The one exception worth flagging for the Orchestrator: `ModelSelectionDecisionReference.decision_fingerprint` (§5, INTEG-01) is entirely caller-constructed with no canonical source — whatever the future Orchestrator's adapter chooses to put there (one of the 5 real fingerprints, or a fresh `stablePayload(decision)`) becomes the de facto standard, since nothing today validates it against the original decision.

**What the Orchestrator should verify vs. transport:** verify (recompute and compare) any fingerprint it is about to register into a registry with an `expected_fingerprint` guard (§10) — the registry does this automatically, but the Orchestrator's own pre-flight check can fail faster with a clearer error. Transport-only (never recompute) every fingerprint embedded inside an already-validated decision/result object it is merely relaying onward (e.g., `section_fingerprints` inside a `ContextAssemblyResult` being handed to a Tool/Workflow layer) — recomputing these would require access to the original section objects, which the Orchestrator should never receive per the "no real content" invariant.

**Terminology check:** grepped every PR #87–89 doc (`HERMES_CONTEXT_ASSEMBLY_ENGINE.md`, `HERMES_TOOL_CONTRACTS.md`, `HERMES_WORKFLOW_CONTRACTS.md`) for "hash" — zero matches; all three explicitly state the canonical-JSON-not-a-hash fact in their own "Fingerprints" section, consistent with PRs #83/#84's docs and the PR85 audit's finding. No documentation/code divergence found on this point across all 9 domains.

## 10. Sequências e Versões

Confirmed by direct grep of `logical_sequence`/`registry_version`/`expected_version`/`expected_fingerprint` across all 8 decision files plus their corresponding request/contract objects:

- **`logical_sequence` belongs to the request/contract, not the decision**, in every domain except one: `context-assembly-request.js` (#87) and `workflow-contract.js` (#89) both carry `logical_sequence` as a field on the *input* object; `agent-policy-decision.js` (#80) is the sole exception, carrying it directly on the *output* decision instead. The other 6 decision files have no `logical_sequence` anywhere. **Recommendation:** the Orchestrator should treat `logical_sequence` as belonging to the request/contract layer uniformly (matching 7 of 8 domains, including both PR #87 and #89) and should not expect to read it back off a decision object except for Agent Policy.
- **`registry_version` belongs to the decision, present in the 5 oldest domains (#80–84), absent in the 3 newest (#87–89).** The 3 newest domains instead rely purely on the registry's own `expected_version` optimistic-concurrency option passed at *write* time (confirmed present and tested in `context-assembly-registry.js`/`tool-registry.js`/`workflow-registry.js`, identical `resolveRegistration` mechanism across all 9 registries per §7) — they simply never echo a version number back onto the decision object itself. **Recommendation:** the Orchestrator should not expect `registry_version` on a Context Assembly/Tool/Workflow decision; if it needs to know what version was stored, it must read that from the registry's own `register*()` return value (`{ version, ... }`, confirmed present in all 9 registries' resolution objects) at the moment of registration, not from the decision later.
- **`expected_version`/`expected_fingerprint` belong to the registry call, uniformly**, across all 9 registries that support them (all except `agent-registry.js`/`agent-policy-registry.js` for `expected_fingerprint`, per CAND-REG-01 §7) — this is the one sequence/version concept that *is* already fully portable across (almost) every domain.
- **Recommendation for the future Orchestrator:** generate its own `logical_sequence` for its own request object (following the #87/#89 convention), pass `expected_version`/`expected_fingerprint` uniformly to every registry call it makes (accepting that 2 of 9 registries silently ignore `expected_fingerprint`), and never assume a decision object it receives back carries either a sequence number or a registry version — both must be tracked by the Orchestrator itself at the point of registration, not recovered from a decision after the fact.

## 13. Integração Tool/Workflow

`workflow-step-contract.js` imports `TOOL_CAPABILITIES` **directly** from `tool-capability-contract.js` (confirmed: `const { TOOL_CAPABILITIES } = require('./tool-capability-contract');`) — `required_capabilities` on a Workflow step and `capabilities` on a Tool's `CapabilitySet` are the exact same 11-value enum, imported once, not independently redeclared. Confirmed no field on `WorkflowStep` implies execution: `tool_reference`/`model_reference`/`context_reference` are nullable minimal `SINGLE_REFERENCE` pointers (`reference_id`/`reference_version`/`reference_fingerprint`/`reference_present`), never a callable handle; `approval_required`/`parallelizable`/`optional` are plain declarative booleans; `estimated_cost_minor_units`/`estimated_duration_ms` are self-declared estimates, never measured.

**Missing information for a future Orchestrator to plan safely, all confirmed absent by direct grep of `workflow-decision.js` (zero matches for each field name checked):**

- **INTEG-04 (P2):** No cross-check that a step's `tool_reference`/`model_reference`/`context_reference` actually resolves to a real, registered `Tool`/`ModelSelectionDecision`/`ContextAssemblyResult` — by design (PRs #88/#89 never auto-query a registry), but this means **nothing today verifies a step's `required_capabilities` are actually a subset of what its referenced tool declares**. - **INTEG-06 (P3):** `WorkflowStep.estimated_cost_minor_units` is independently author-declared with no cross-check against the referenced Tool's actual `cost_tier` (`tool-cost-contract.js`).
- **INTEG-07 (P3):** `step.parallelizable` (boolean) and a `PARALLEL_REFERENCE`-typed entry in `depends_on` are two independent signals for the same concept, never cross-validated for consistency with each other.
- **INTEG-03 (P2, restated from §4 scenario 10):** `approval_required=true` on one or more steps never surfaces as a distinct `WorkflowDecision.status` value — `WORKFLOW_REGISTERED_SIMULATION` means "structurally valid," not "clear to run without a human." An Orchestrator must independently walk every step's `approval_required` flag.

None of INTEG-03 through INTEG-07 are contract bugs — every one is a deliberate consequence of PRs #88/#89's own "no automatic registry consultation" design. They are exactly the gaps the future Orchestrator exists to close, and are recorded here so that PR is not designed as if today's contracts already close them.

## 17. Correções Recomendadas

Ordered by what actually blocks or meaningfully de-risks the future Orchestrator PR, not by finding ID:

1. **Write the `ModelSelectionDecision` → `ModelSelectionDecisionReference` adapter (INTEG-01).** The single concrete piece of new code this audit recommends — a small, named, tested function, not inline mapping at every call site. Required before any real Model Selection → Context Assembly composition works.
2. **Add the two missing cross-checks between Tool and Workflow (INTEG-04, INTEG-05)** — capability-subset and side-effect-vs-approval/compensation. Both are naturally the Orchestrator's own responsibility (per its "cross-domain composition" charter), not a fix to PRs #88/#89 themselves, which are correctly scoped to never auto-query a registry.
3. **Add an approval/human-review aggregation step (INTEG-02, INTEG-03)** that walks `Workflow.steps[].approval_required` and `ModelSelectionTaskProfile.human_review_required` and produces one clear "needs human sign-off" summary — neither signal surfaces on a decision `status` today, by design, and both are the same class of gap.
4. **Fix `agent-registry.js`/`agent-policy-registry.js`'s missing `FINGERPRINT_CONFLICT` support (CAND-REG-01)** — small, isolated, same shape as PR #86's `ORGANIZATION_BLOCKED` fix for the same two files. Recommended before the Orchestrator starts passing `expected_fingerprint` uniformly to all 9 registries.
5. **Document, don't fix, everything else** — INTEG-06/07/08/11/12/13/14/15, CAND-REG-02/03/04, CAND-AUD-01/02/03, DEPS-90-1/90-2 are all either intentional architectural evolution, correct-by-contract-shape, or low-value/high-risk-to-touch right now. All are captured in `HERMES_AGENT_ORCHESTRATOR_READINESS.md` so the next PR is designed with them in mind rather than rediscovering them mid-implementation. None require a code change in PRs #79–89 themselves.

Items 1–3 are the only ones this audit considers meaningful blockers to a clean Orchestrator implementation; item 4 is small and cheap enough to bundle in either before or alongside the Orchestrator PR without materially delaying it. No item requires touching a contract's field list, enum, or fingerprint computation.
